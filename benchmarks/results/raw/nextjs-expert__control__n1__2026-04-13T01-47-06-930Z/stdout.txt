**How the OCC works:** `.update({ version: expectedVersion + 1 }).eq('version', expectedVersion)` translates to `UPDATE posts SET version = N+1 WHERE id = $1 AND version = N`. Two concurrent requests with the same `expectedVersion` will race — only one gets the row lock and succeeds; the other matches 0 rows. `.maybeSingle()` returns `data: null` (no error) on 0 matched rows, which the `!data` check maps to `'version-conflict'`.

**Note on `like_count` increment:** Supabase's `.update()` / PostgREST PATCH doesn't support SQL column expressions (`col = col + 1`). In production you'd use a database function via `supabase.rpc()` or a `BEFORE UPDATE` trigger to atomically increment `like_count`. The version-check guard here is the core OCC mechanism that prevents lost updates.
