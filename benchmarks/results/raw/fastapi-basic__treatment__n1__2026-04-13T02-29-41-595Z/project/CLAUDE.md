# Project Harness — FastAPI + PostgreSQL

This project uses a development harness generated by `harness-marketplace`. When performing any task, follow the pipeline and respect the hook enforcement.

## Pipeline

Every non-trivial task goes through three phases:

**1. Plan** — Before editing code, briefly outline:
- What endpoints/models/schemas will change
- Why each change is needed
- What could break (migrations, clients, tests)

**2. Implement** — Make changes incrementally:
- One logical change at a time
- SQLAlchemy 2.0 typed syntax (`Mapped[T]`, `mapped_column`) — not legacy Column()
- Use dependency injection (`Depends(get_db)`) for DB sessions in routes
- If a hook blocks an action, address the underlying issue rather than working around it

**3. Verify** — After implementation, check:
- Tests pass (`pytest`)
- Lint clean (`ruff check .`)
- Types clean (`mypy app`) if feasible
- No hardcoded secrets or credentials
- No edits to protected files (`.env*`)
- Database queries are parameterized (never f-string SQL)

## Hook Enforcement (active)

| Hook | Event | What it enforces |
|------|-------|------------------|
| protected-files | PreToolUse(Edit/Write) | Blocks edits to `.env*`, lock files, alembic migrations |
| secret-guard | PreToolUse(Write) | Blocks hardcoded secrets (API keys, JWT secrets, DB creds) |
| pattern-guard | PreToolUse(Edit) | Architecture rules (routes must not execute raw SQL; use SQLAlchemy) |
| db-safety | PreToolUse(Bash) | Blocks dangerous SQL (DROP TABLE, TRUNCATE, DELETE without WHERE) |

## Stack Conventions

- FastAPI with explicit `Depends()` for injection
- SQLAlchemy 2.0 typed models (`from sqlalchemy.orm import Mapped, mapped_column`)
- Pydantic v2 schemas separate from ORM models
- Routes live in `app/routes/`, models in `app/models/`, schemas in `app/schemas/`
- Settings via `pydantic-settings` from `app/settings.py`
- Tests use `fastapi.testclient.TestClient` + pytest fixtures

## When in Doubt

- Prefer simple, working implementations
- Write tests when the task requires verification
- Never commit .env files or inline secrets
- Parameterize all SQL — never interpolate user input into query strings
- Ask for clarification if requirements are ambiguous — do not guess
