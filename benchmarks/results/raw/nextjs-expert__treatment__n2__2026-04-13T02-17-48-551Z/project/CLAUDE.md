# Project Harness — Next.js + Supabase

This project uses a development harness generated by `harness-marketplace`. When performing any task, follow the pipeline and respect the hook enforcement.

## Pipeline

Every non-trivial task goes through three phases:

**1. Plan** — Before editing code, briefly outline:
- What files will change
- Why each change is needed
- What could break

**2. Implement** — Make changes incrementally:
- One logical change at a time
- Run `npm run lint` and `npm run typecheck` mentally as you go
- If a hook blocks an action, address the underlying issue rather than working around it

**3. Verify** — After implementation, check:
- Acceptance criteria met (build, lint, typecheck pass if feasible)
- No hardcoded secrets or credentials
- No edits to protected files (`.env*`, lock files)
- Supabase queries use parameterized inputs (never string concatenation)

## Hook Enforcement (active)

The following hooks are active and will block/warn on violations:

| Hook | Event | What it enforces |
|------|-------|------------------|
| protected-files | PreToolUse(Edit/Write) | Blocks edits to `.env*`, `package-lock.json`, `yarn.lock`, `pnpm-lock.yaml` |
| secret-guard | PreToolUse(Write) | Blocks hardcoded secrets (API keys, tokens, private keys) |
| pattern-guard | PreToolUse(Edit) | Architecture rules (currently: no direct DB imports in API handlers) |
| db-safety | PreToolUse(Bash) | Blocks dangerous SQL (DROP TABLE, TRUNCATE, DELETE without WHERE) |
| post-edit-lint | PostToolUse | Runs ESLint after every edit (warning, non-blocking) |

## Stack Conventions

- Next.js 14 App Router (use `app/` directory, not `pages/`)
- TypeScript strict mode
- Supabase client from `lib/supabase.ts` (do not instantiate new clients inline)
- Environment variables: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- Server components by default; use `'use client'` only when needed for interactivity

## When in Doubt

- Prefer minimal, working implementations over speculative abstractions
- Write tests when the task requires verification
- Never commit .env files or inline secrets
- Ask for clarification if requirements are ambiguous — do not guess
