**Files created:**

- `app/models/post.py` — SQLAlchemy 2.0 `Post` model with `id`, `user_id` (FK), `title` (String 200), `body` (Text), `created_at` (server default), and `user: Mapped["User"]` relationship.
- `app/schemas/__init__.py` — empty package marker.
- `app/schemas/user_with_posts.py` — `PostResponse` and `UserWithPostsResponse` Pydantic v2 models with `from_attributes = True`.
- `app/routes/users_with_posts.py` — `GET /users-with-posts` using `selectinload(User.posts)` so a single `SELECT users` + one `SELECT posts WHERE user_id IN (...)` is issued instead of N+1.
- `tests/test_users_with_posts.py` — two tests: one checks response structure, one uses SQLAlchemy's `before_cursor_execute` engine event to assert ≤2 SELECT queries are fired even with 5 users (vs 6 with N+1).

**Files modified:**

- `app/models/user.py` — added `posts: Mapped[list["Post"]] = relationship(back_populates="user")`.
- `app/main.py` — registered `users_with_posts.router`.

**Key design decisions:**

- `selectinload` over `joinedload`: avoids cartesian-product row duplication when users have many posts, and issues exactly 2 queries.
- `Annotated[Session, Depends(get_db)]` alias avoids ruff B008 (function call in default argument).
- `StaticPool` in tests ensures all SQLite in-memory operations share one connection (without it, `create_all` and the test session silently use separate empty databases).
