# Benchmarks — Phase 0 A/B Pilot

Empirical validation that harnesses generated by `harness-marketplace` improve Claude Code output quality compared to running without a harness.

## Background

Inspired by `revfactory/claude-code-harness` (which reported +60% average quality improvement across 15 tasks with a fixed harness), this benchmark adapts the methodology for our project's key difference: **our harness is dynamically generated per-project** rather than fixed.

We therefore can't test "the harness" as a single artifact. Instead we test **"what a harness generated by our wizard produces"** on specific reference project configurations.

## Phase 0 Scope (Multi-Stack Pilot)

**Why multi-stack**: Testing on a single stack risks overfitting the conclusion to one ecosystem (e.g., "works for Next.js but not Python"). Two anchor stacks give early signal on generalization without full Phase 1 cost.

| Stack | Purpose | Harness coverage |
|-------|---------|-----------------|
| Next.js + TypeScript + Supabase | Web/full-stack anchor | Max — protected files, pattern guard, db-safety, secret guard, format/lint/typecheck |
| Python FastAPI + PostgreSQL | Backend anchor | Protected files, db-safety, secret guard, pattern guard, format/lint |

### Experimental design

| Dimension | Value |
|-----------|-------|
| Reference projects | 2 (Next.js+Supabase, FastAPI+Postgres) |
| Tasks per stack | 3 (Basic / Advanced / Expert) |
| Conditions | control (no harness) vs treatment (harness installed) |
| N (runs per cell) | 2 |
| **Total runs** | **24** |

### Measurement

**1. Automated metrics** (`scorer/automated.js`)
- File existence / acceptance criteria
- Build success (`npm run build` / `pytest --collect-only`)
- Lint / typecheck pass
- Hook-block event count (treatment only)

**2. LLM judge** (`scorer/llm-judge.js`)
Blind scoring on 4 dimensions (1-10):
- Code quality (readability, structure)
- Completeness (requirements coverage)
- Edge case handling
- Security (obvious vulnerabilities)

**3. Differential harness metrics** (treatment-specific)
- Hook blocks triggered
- Pipeline structure adherence

## How a Run Works

```
node benchmarks/runner/run.js \
  --task nextjs-basic \
  --condition control \
  --run 1
```

Under the hood:

1. Copy `reference-projects/nextjs-supabase-seed/` to `/tmp/bench-{run-id}/`
2. If `--condition treatment`, overlay `nextjs-supabase-harness/` on top (adds `.claude/`, `settings.json`)
3. Initialize git in the temp dir
4. Invoke `claude -p "<task prompt>" --cwd /tmp/bench-{run-id}/ --model sonnet`
5. Capture stdout/stderr, final file state, hook block log
6. Save to `results/raw/{run-id}/`
7. Cleanup temp dir (optional)

## How Scoring Works

```
node benchmarks/scorer/automated.js --run {run-id}
node benchmarks/scorer/llm-judge.js --run {run-id}
```

Automated scorer runs build/lint/typecheck in the captured project state, parses exit codes, counts hook block events from `hook-blocks.log`.

LLM judge reads the final file state, compares against acceptance criteria from the task spec, assigns 1-10 scores per rubric dimension. Runs with `--model sonnet` for cost control.

## Executing the Full Pilot

```bash
# 1. Verify infrastructure (sanity run)
node benchmarks/runner/run.js --task nextjs-basic --condition control --run sanity

# 2. Execute 24 runs in background
bash benchmarks/runner/batch.sh

# 3. Score all runs
bash benchmarks/scorer/score-all.sh

# 4. Generate report
node benchmarks/scorer/aggregate.js > benchmarks/results/phase0-report.md
```

## Interpreting Results

Phase 0 is **directional**, not publishable. Expected outcomes:

- **Strong positive signal (+20% or more)**: proceed to Phase 1 with confidence
- **Weak positive signal (+5-20%)**: proceed to Phase 1 with refined methodology
- **No signal**: investigate why (hooks not firing? tasks too easy? variance too high?) before Phase 1
- **Negative signal**: investigate whether harness is actively slowing things down (unexpected)

See `results/phase0-report.md` after execution for actual numbers and Phase 1 recommendation.

## Scientific Caveats

Phase 0 is a pilot, not a controlled study. Limitations:

- N=2 per cell is below statistical power threshold
- Two reference projects don't generalize to all project types
- LLM variance may dominate signal
- Scoring involves subjective dimensions (even with rubric)
- Runner uses parent Claude Code auth (OAuth), not a clean API-key setup — may introduce session-state carryover

Phase 1 would address these by pre-registering methodology, increasing N, adding statistical tests, and using independent API-key subprocesses.
