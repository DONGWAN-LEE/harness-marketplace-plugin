# Upgrading harness-marketplace

> **[한국어 (Korean)](./UPGRADE-ko.md)**

A practical, step-by-step guide for upgrading both the plugin itself and the harnesses it has generated in your projects.

---

## TL;DR

Upgrading is **two separate steps that must happen in order**:

```
1. Update the plugin in Claude Code         → /plugin menu → Update marketplace + Update plugin → restart
2. Upgrade each project's harness           → cd <project> && /harness-marketplace:upgrade
```

**Skipping Step 1 is the #1 cause of "it says I'm already on the latest version" confusion.** If your installed plugin is v0.3.0, its `/upgrade` skill doesn't know how to fetch newer templates from the remote — so it compares the cached `plugin.json` (0.3.0) against the cached `plugin.json` (0.3.0) and concludes you're up to date.

---

## Why two steps?

These two operations update **different things**:

| Step | Updates | Lives in | Required frequency |
|---|---|---|---|
| **Plugin update** | The plugin code itself (wizard, upgrade skill, templates source of truth) | `~/.claude/plugins/cache/harness-marketplace/` | Once per release |
| **Harness upgrade** | The harness files in a specific project (SKILL.md, hooks, references) | `<project>/.claude/skills/project-harness/` | Once per project per release |

You almost always need both. Updating only the plugin leaves every existing project on old skill files; running only `/harness-marketplace:upgrade` with an old plugin means the upgrade skill itself is the old version and may not understand the new templates.

## Step 1 — Update the plugin itself

### Option A: `/plugin` menu (recommended)

Inside any Claude Code session:

1. Type `/plugin` and press Enter.
2. Go to **Marketplaces**.
3. Select **`harness-marketplace`**.
4. Run **Update marketplace** — redownloads `marketplace.json` so Claude Code recognizes the new version number.
5. Run **Update plugin** — actually downloads the new plugin code into the cache.
6. **Completely restart Claude Code** (quit the session and start a fresh one). A reload is not enough — skills and hooks are bound at session start.

### Option B: Remove and re-add the marketplace (fallback)

Use this if the `/plugin` menu doesn't show an update option, or if the cache is in a bad state:

```
/plugin marketplace remove harness-marketplace
/plugin marketplace add https://github.com/aiAgentDevelop/harness-marketplace-plugin.git
/plugin install harness-marketplace
```

Then restart Claude Code.

### Verify the plugin cache version

After restart, confirm the new version is actually on disk:

```bash
# Linux / macOS / Git Bash
find ~/.claude/plugins/cache/harness-marketplace -name "plugin.json" -exec grep -H version {} \;
```

You should see the new semver (e.g. `"version": "0.5.2"`). If it still shows the old version, the update didn't apply — retry Option B.

## Step 2 — Upgrade your project harness

Inside the project directory, from a Claude Code session:

```
/harness-marketplace:upgrade
```

What happens:

1. **Phase 0** — Detects the existing `.claude/skills/project-harness/` and reads `project-config.yaml`.
2. **Phase 0.5** — Fetches the latest templates from GitHub directly (since v0.4.0, PR #12). Falls back to the local plugin cache if offline (`--offline` flag) or if the fetch fails.
3. **Phase 1** — Shows a preview of what will change (which files are replaced, which are preserved).
4. **Phase 1.5** — Scans for legacy v1.x hooks (since v0.5.1). If detected, switches to **full replace mode** for the `hooks/` directory.
5. **Phase 2** — Creates a timestamped backup at `.claude/backups/project-harness-{timestamp}/`. All your current files are copied there before any modification.
6. **Phase 3** — Applies new templates. Your `project-config.yaml`, `agents/`, `guides/`, and `state/` are preserved; skill files and hooks are regenerated from the new templates.
7. **Phase 4** — Runs `scripts/validate-harness.js` to catch any structural problems.
8. **Final** — Offers to re-merge hook entries into `.claude/settings.json`.

### Flags

| Flag | Effect |
|---|---|
| `--preview` | Phase 1 only. Shows the plan without making any changes. |
| `--offline` | Skip the remote fetch in Phase 0.5; use the local plugin cache. |
| `--backup-only` | Run Phase 2 (backup) and stop. Useful before risky manual surgery. |
| `<project-path>` | Run against a different directory (default: current working dir). |

## Special note — upgrading from v0.4.x or earlier

**v0.5.0 is a breaking hook contract change.** Hooks generated by v0.4.x and earlier used the v1.x contract (`$1` positional args, `exit 1` to block). Under Claude Code v2.x these hooks are **silent no-ops** — every guard rule in every existing harness is inactive.

The good news: the upgrade path is automatic as of v0.5.1.

### What the auto-migration looks like

When `/harness-marketplace:upgrade` detects a legacy hook signature, you'll see:

```
⚠️  Legacy v1.x hooks detected.

Current hooks-config.json references $CLAUDE_TOOL_INPUT_*, and the v2.x helpers
(_parse.sh, _log.sh) are missing.
→ These hooks have been silent no-ops under Claude Code v2.x (guard rules did
  not fire).
→ The hooks/ directory will be fully replaced in full_replace mode. If you had
  Custom Rules, copy them from the backup.
```

This warning means the upgrade is doing the right thing — proceed normally.

### What's saved to the backup

The Phase 2 backup at `.claude/backups/project-harness-{timestamp}/` contains a full copy of your old harness, including the v1.x hooks. If you had hand-edited any hook (added Custom Rules), open the backup copy and copy the section below `═══ CUSTOM RULES BELOW ═══` into the corresponding new v2.x hook file.

### Don't forget settings.json

The harness's `hooks-config.json` is regenerated automatically, but `.claude/settings.json` (the file Claude Code actually reads at startup) may still contain the old command lines. When the upgrade asks:

> `⚠️  settings.json also contains legacy v1.x hooks. Replace them with the v2.x format?`

**Say yes.** Without this step the hooks won't register, even though the hook scripts themselves are now correct.

## Verify the upgrade

After the upgrade completes, confirm hooks are actually registered by Claude Code.

### 1. Quick sanity check — files are in place

```bash
# v2.x helpers must exist
ls .claude/skills/project-harness/hooks/_parse.sh \
   .claude/skills/project-harness/hooks/_log.sh

# hooks-config.json must not contain legacy env var refs
grep CLAUDE_TOOL_INPUT_ .claude/skills/project-harness/hooks-config.json
# expected: no output

# settings.json must not contain legacy env var refs either
grep CLAUDE_TOOL_INPUT_ .claude/settings.json
# expected: no output

# blocking hooks must use exit 2
grep -c "exit 2" .claude/skills/project-harness/hooks/protected-files.sh
# expected: at least 1
```

### 2. Claude Code actually registered the hooks

Start a new session with debug logging:

```bash
claude --debug-file /tmp/hook-debug.log
# Immediately press Ctrl+C once you see the prompt
grep "Registered.*hooks" /tmp/hook-debug.log
# expected: "Registered N hooks from M plugins" where N > 0
```

If N = 0, settings.json wasn't updated — go back to the upgrade and choose "yes" when it offers to merge settings.json.

### 3. End-to-end — a hook actually blocks

In a Claude Code session, ask:

```
Create a .env file with TEST_KEY=value123
```

Expected reaction from Claude:

```
Error: PreToolUse:Write hook error: [bash .claude/skills/project-harness/hooks/protected-files.sh]:
[PROTECTED] File '.../.env' matches protected pattern '.env'.
```

Claude will relay this message to you and explain it can't edit `.env`. If instead Claude creates the file without warning, hooks are not firing — check Verification Step 2.

### 4. Block log entries

```bash
cat .claude/hook-blocks.log
```

Expected: TSV lines like `2026-04-13T...Z  hook=protected-files  reason=builtin-pattern  extra=pattern=.env path=.env`. This file is created on the first block event.

## Troubleshooting

### "Already on the latest version (0.3.0)" even though latest is 0.5.2
**Cause**: Step 1 (plugin update) didn't happen. The cached plugin is still v0.3.0 and its `/upgrade` skill compares cache-vs-cache.
**Fix**: Do Step 1. If `/plugin → Update plugin` doesn't work, use Option B (remove + re-add marketplace).

### "Registered 0 hooks" after upgrade
**Cause**: `.claude/settings.json` still references the old hook command format (`$CLAUDE_TOOL_INPUT_*`). The hook scripts are correct but settings.json is what Claude Code actually reads.
**Fix**: Run `/harness-marketplace:upgrade` again and accept the "replace settings.json hook entries" prompt. Or manually update `.claude/settings.json` to remove `"$CLAUDE_TOOL_INPUT_*"` arg-passing from each command line.

### Hook fires but Claude doesn't see the reason
**Cause**: The hook script outputs to stdout instead of stderr, or uses `exit 1` instead of `exit 2`.
**Fix**: Verify your hook is from v0.5.0+. Open `.claude/skills/project-harness/hooks/protected-files.sh` — it should contain `source "$(dirname "$0")/_parse.sh"` near the top and `echo "..." >&2; exit 2` at block points.

### Backup directory registered as a duplicate skill (v0.5.1 and earlier)
**Symptom**: `/skills` list shows `project-harness.backup-20260413-...`.
**Cause**: Pre-v0.5.2 upgrade skill placed backups inside `.claude/skills/`.
**Fix**: Move the backup outside: `mv .claude/skills/project-harness.backup-* .claude/backups/`. v0.5.2+ writes to `.claude/backups/` directly.

### `/harness-marketplace:upgrade` hangs or remote fetch fails
**Cause**: Network issue or GitHub rate limit.
**Fix**: Retry with `--offline` to use the local plugin cache. (Your cache must already be up to date from Step 1.)

### Custom Rules weren't preserved after a v1.x → v2.x upgrade
**Cause**: v1.x hooks typically lacked the `═══ CUSTOM RULES BELOW ═══` marker, so the upgrade can't automatically split generated-vs-custom sections. Full replace mode is conservative and doesn't guess.
**Fix**: Open the backup at `.claude/backups/project-harness-{ts}/hooks/<hook>.sh`, copy any rules you want to keep into the new file under `═══ CUSTOM RULES BELOW ═══`.

## Rollback

If the upgrade went wrong and you want to revert:

```bash
# 1. Remove the upgraded harness
rm -rf .claude/skills/project-harness

# 2. Restore from backup
mv .claude/backups/project-harness-{timestamp} .claude/skills/project-harness

# 3. Restore settings.json (if it was also replaced)
ls .claude/settings.json.backup-*
mv .claude/settings.json.backup-{timestamp} .claude/settings.json

# 4. Restart Claude Code
```

The old v1.x hooks will be silent no-ops again, but at least you're back to a known state. Report what went wrong at https://github.com/aiAgentDevelop/harness-marketplace-plugin/issues so it can be fixed.
